# autoresearch

這是一個讓 LLM 自行進行研究的實驗。

## 初始設置

若要建立新實驗，請與使用者協作完成以下步驟：

1. **確認執行標籤**：依據今天的日期提出一個標籤（例如 `mar5`）。分支 `autoresearch/<tag>` 必須尚不存在——這是一次全新執行。
2. **建立分支**：從當前 master 執行 `git checkout -b autoresearch/<tag>`。
3. **讀取範疇內的檔案**：此 repo 規模精簡。請閱讀以下檔案以取得完整上下文：
   - `README.md` — 儲存庫背景說明。
   - `prepare.py` — 固定常數、資料準備、Tokenizer、Dataloader、Evaluation。**請勿修改。**
   - `train.py` — 你所修改的檔案。模型架構、最佳化器、訓練迴圈。
4. **確認資料存在**：檢查 `~/.cache/autoresearch/` 是否包含資料分片與 Tokenizer。若無，請告知使用者執行 `uv run prepare.py`。
5. **初始化 results.tsv**：建立僅含標題列的 `results.tsv`。基準值將在第一次執行後記錄。
6. **確認並開始**：確認設置無誤。

獲得確認後，開始進行實驗。

## 實驗

每次實驗在單張 GPU 上執行。訓練腳本以 **固定 5 分鐘的時間預算** 執行（掛鐘訓練時間，不含啟動/編譯時間）。啟動方式為：`uv run train.py`。

**你可以做的事：**
- 修改 `train.py`——這是你唯一可以編輯的檔案。一切皆為公平競技場：模型架構、最佳化器、超參數、訓練迴圈、Batch Size、模型大小等。

**你不能做的事：**
- 修改 `prepare.py`。它是唯讀的。其中包含固定的 Evaluation、資料載入、Tokenizer 及訓練常數（時間預算、序列長度等）。
- 安裝新套件或新增相依。你只能使用 `pyproject.toml` 中已有的套件。
- 修改 Evaluation 框架。`prepare.py` 中的 `evaluate_bpb` 函式是最終的基準指標。

**目標很簡單：讓 val_bpb 越低越好。** 由於時間預算固定，你無需擔心訓練時間——永遠是 5 分鐘。一切皆可調整：架構、最佳化器、超參數、Batch Size、模型大小。唯一的限制是程式碼能夠正常執行且在時間預算內完成。

**VRAM** 是軟性限制。為了有意義的 val_bpb 提升，少量增加是可接受的，但不應大幅暴增。

**簡潔性原則**：在其他條件相同的情況下，越簡潔越好。若一個微小改善引入了醜陋的複雜度，不值得保留。反之，刪除某些東西並獲得相同或更好的結果，是很棒的成果——這是簡化上的勝利。評估是否保留改動時，請權衡複雜度成本與改善幅度。一個提升 0.001 val_bpb 但新增 20 行臨時程式碼的改動？可能不值得。一個透過刪除程式碼提升 0.001 val_bpb 的改動？絕對保留。改善幾乎為零但程式碼大幅簡化？保留。

**第一次執行**：你的第一次執行必須是建立基準，因此請直接執行訓練腳本，不做任何修改。

## 輸出格式

腳本完成後會印出如下摘要：

```
---
val_bpb:          0.997900
training_seconds: 300.1
total_seconds:    325.9
peak_vram_mb:     45060.2
mfu_percent:      39.80
total_tokens_M:   499.6
num_steps:        953
num_params_M:     50.3
depth:            8
```

請注意，腳本設定為固定在 5 分鐘後停止，因此依據此電腦的運算平台不同，數字可能有所差異。你可以從日誌檔案中提取關鍵指標：

```
grep "^val_bpb:" run.log
```

## 記錄結果

實驗完成後，將結果記錄至 `results.tsv`（以 Tab 分隔，**不是**逗號分隔——逗號在描述中會造成問題）。

TSV 包含一個標題列與 5 個欄位：

```
commit	val_bpb	memory_gb	status	description
```

1. git commit 雜湊值（短版，7 個字元）
2. 達到的 val_bpb（例如 1.234567）——崩潰時填 0.000000
3. 峰值記憶體（GB），四捨五入至小數點後 1 位（例如 12.3——將 peak_vram_mb 除以 1024）——崩潰時填 0.0
4. 狀態：`keep`（保留）、`discard`（捨棄）或 `crash`（崩潰）
5. 本次實驗嘗試內容的簡短文字說明

範例：

```
commit	val_bpb	memory_gb	status	description
a1b2c3d	0.997900	44.0	keep	baseline
b2c3d4e	0.993200	44.2	keep	increase LR to 0.04
c3d4e5f	1.005000	44.0	discard	switch to GeLU activation
d4e5f6g	0.000000	0.0	crash	double model width (OOM)
```

## 實驗迴圈

實驗在專屬分支上執行（例如 `autoresearch/mar5` 或 `autoresearch/mar5-gpu0`）。

無限迴圈：

1. 查看 git 狀態：目前所在的分支/commit
2. 直接修改程式碼，以實驗性想法調整 `train.py`
3. git commit
4. 執行實驗：`uv run train.py > run.log 2>&1`（重新導向所有輸出——**不要**使用 tee 或讓輸出淹沒你的 context）
5. 讀取結果：`grep "^val_bpb:\|^peak_vram_mb:" run.log`
6. 若 grep 輸出為空，表示執行崩潰。執行 `tail -n 50 run.log` 讀取 Python 堆疊追蹤並嘗試修復。若多次嘗試仍無法解決，放棄該實驗。
7. 將結果記錄至 TSV（**注意：不要 commit results.tsv 檔案，讓它保持 git 未追蹤狀態**）
8. 若 val_bpb 改善（更低），則「推進」分支，保留該 git commit
9. 若 val_bpb 相同或更差，則 git reset 回到起始點

核心思路是：你是一位完全自主的研究者，不斷嘗試各種方法。有效就保留，無效就捨棄。你在推進分支以便迭代。若感覺陷入瓶頸，可以回退，但應極為謹慎（甚至從不這樣做）。

**逾時**：每次實驗總計應耗時約 5 分鐘（加上數秒的啟動與 Evaluation 額外開銷）。若執行超過 10 分鐘，終止它並視為失敗（捨棄並還原）。

**崩潰**：若執行崩潰（OOM、Bug 等），請自行判斷：若是簡單易修的問題（例如拼寫錯誤、缺少 import），修復後重新執行。若想法本身根本上有問題，直接跳過，在 TSV 中記錄狀態為 `crash`，然後繼續。

**永不停止**：實驗迴圈一旦開始（初始設置完成後），**絕對不要**暫停詢問使用者是否應繼續。**不要**問「要繼續嗎？」或「這是個好的停止點嗎？」。使用者可能正在睡覺，或已離開電腦，並期望你**無限期**持續工作，直到被手動停止。你是自主的。若想法耗盡，請更努力思考——閱讀程式碼中引用的論文、重新閱讀範疇內的檔案以尋找新角度、嘗試結合先前的接近成功案例、嘗試更激進的架構變更。迴圈持續執行，直到使用者中斷你，句點。

舉例來說，使用者可能在睡覺時讓你持續執行。若每次實驗耗時約 5 分鐘，你每小時可執行約 12 次，一般人睡眠期間共約 100 次。使用者醒來後，便能看到所有由你在他們睡眠期間完成的實驗結果！
